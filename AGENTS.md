# Volunteer Hub — хакатон (дедлайн 20.07 09:00, UTC+5)

## Контекст (НЕ переспрашивать пользователя, работать автономно)

Хакатон «Startup Platform для партнёра», соло. Партнёр выбран: **Employee Volunteer Program (NVS & Zhurekten Zhurekke)** — программа корпоративного волонтёрства Chevron в Казахстане (субботники, посадка деревьев, поддержка детей с инвалидностью и из малообеспеченных семей; города: Астана, Алматы, Атырау).

**Продукт:** Volunteer Hub — веб-приложение, где координатор публикует волонтёрские события, а сотрудники записываются на них в 1 клик. Боль: сейчас записи на такие акции ведутся в рассылках/Excel, координатор не видит, кто и сколько записался.

**Критерии хакатона:** работающий продукт (40%), релевантность сфере партнёра (25%), фидбэк 5+ реальных пользователей (20%), питч (15%).

## Стек (ничего не добавлять!)

- Один статический сайт: `index.html` + `app.js` + `config.js`. Без фреймворков, без сборки, без npm-пакетов в рантайме.
- Supabase через CDN: `<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2"></script>` (глобальный `window.supabase`).
- БД: Supabase (схема в `schema.sql`, пользователь уже применил её и вставил ключи в `config.js`).
- Деплой: Vercel, статика, `npx vercel --prod --yes`.
- UI на русском. Стили — один `<style>` в index.html, минимально прилично.

## Функционал (всё, больше ничего)

1. Список событий из таблицы `events`: название, город, дата, категория, описание, сколько записалось (`signups`), кнопка «Записаться».
2. Запись на событие: имя + контакт → insert в `signups`. После записи обновить список.
3. «Я координатор»: кнопка открывает форму добавления события; поле «код организатора», сверка со строкой `zhurek2026` в app.js. Неверный код — alert, не сохранять. (ponytail: код косметический, спам-защита не нужна на хакатон)
4. Форма «Оставить отзыв о платформе»: имя/роль, оценка 1–5, текст → insert в `feedback`. Это сбор фидбэка для хакатона.

## Задачи — по порядку. После КАЖДОЙ: проверка → если падает, чинить самому → git commit.

1. `index.html` + `app.js` + `config.js`: подключение Supabase, вывод списка событий.
   Проверка: `node test.mjs` — скрипт ниже, писать GET events возвращает строки.
2. Запись на событие (insert в signups, обновление счётчика).
   Проверка: test.mjs POST signup → 201.
3. Форма координатора с кодом. Проверка: открыть страницу через `python3 -m http.server 8080` и curl — страница отдаётся; логику проверить глазами по коду + node-синтаксис `node --check app.js`.
4. Форма отзыва. Проверка: test.mjs POST feedback → 201.
5. `README.md`: партнёр, проблема, стек, запуск (открыть ссылку Vercel / локально `python3 -m http.server`).
6. `FEEDBACK.md`: шаблон таблицы на 5 пользователей (роль, когда тестировал, что сказал, подтверждение, что изменили) + раздел «Выводы». Заполняет пользователь, фальсифицировать нельзя.
7. Деплой: `npx vercel --prod --yes`. Если просит логин — написать пользователю точную команду `npx vercel login` и остановиться, всё остальное должно быть готово.
8. Финальный `node test.mjs` против прод-URL конфига + commit.

`test.mjs`: node-скрипт, читает URL/ключ из `config.js` (через fs+regex, без импорта), делает fetch в Supabase REST (`/rest/v1/events?select=*` с заголовками apikey/Authorization), assert что статус 200 и массив непустой; POST signup/feedback с `Prefer: return=minimal`, assert 201. Тестовые строки с пометкой "тест" — пользователь удалит в дашборде.

## Правила

- Минимум кода. Никаких новых зависимостей, абстракций, папок, роутеров.
- Не задавать пользователю вопросов до конца списка. Ошибки чинить самому.
- Не ходить в интернет, кроме деплоя.
- Коммиты: `feat: список событий`, `feat: запись на событие` и т.п., после каждой зелёной проверки.
- Если config.js содержит заглушки "ВСТАВЬ" — сразу написать пользователю одну строку: «вставь ключи из Supabase в config.js» — и продолжить всё, что можно без БД.
